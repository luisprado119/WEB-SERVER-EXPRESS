Express Básico

Configuración inicial, servidor y archivos estáticos

¿Qué es Express.js?

Framework minimalista y flexible para Node.js que permite crear servidores HTTP con rutas, middlewares y plantillas. Es el más popular del ecosistema Node.

Instalación

Iniciar un proyecto

# Inicializar package.json
npm init -y
# Instalar express
npm install express

Middlewares

Funciones que procesan cada petición antes de llegar a la ruta

Flujo de un middleware

Petición → Middleware 1 → Middleware 2 → … → Ruta → Respuesta. Cada middleware tiene acceso a (req, res, next) y puede modificar la petición, responder directamente o pasar al siguiente con next().

Morgan

Logging HTTP

Registra cada petición en consola con método, URL, status y tiempo de respuesta.

npm i morgan
const morgan = require('morgan')
app.use(morgan('dev'))

express-fileupload

Subida de ficheros

Intercepta el body multipart/form-data y coloca los ficheros en req.files.

npm i express-fileupload

express.json

Parseo del body

Parsea el body JSON y lo hace disponible en req.body.

app.use(express.json())
app.use(express.urlencoded({
  extended: true

Personalizado

Middleware propio

Cualquier función con (req, res, next). Se ejecuta en cada petición si se monta con app.use().

app.use((req, res, next) => {
  console.log(req.method, req.url)
  next()

404 Not Found

Manejador global para rutas inexistentes

¿Cómo funciona?

El manejador 404 debe colocarse al final de todas las rutas. Express lo alcanza solo cuando ninguna ruta anterior coincidió con la petición. El wildcard app.get('*', ...) captura cualquier ruta no definida.

Opción 1 – Texto plano

Respuesta directa

app.get('*', (req, res) => {
  res.status(404)
     .send('Lo sentimos, no tenemos esto')
})

Opción 2 – Página HTML ✓

Redirección a página de error

app.get('*', (req, res) => {
  res.status(404)
     .redirect('/error_404.html')
})
Mejor UX
GET /ruta-que-no-existe Prueba el manejador 404 personalizado Probar →
GET /error_404.html Página de error 404 con diseño Ver →

Error 500 – Internal Server Error

Middleware de error de Express con 4 parámetros

⚠ Firma de 4 parámetros obligatoria

Express identifica un middleware como manejador de errores solo si tiene exactamente 4 parámetros: (err, req, res, next). Si usas 3, Express no lo tratará como manejador de errores.

Ruta que falla

Lanzar error deliberado

// Ruta de prueba que lanza un error
app.get('/error', (req, res) => {
  throw new Error('esto falla')
})

Middleware de error

Capturar y responder

app.use((err, req, res, next) => {
  console.error(err.stack)
  res.status(500)
     .send(`${err} — 500`)
})
4 parámetros

async/await

Errores asíncronos

app.get('/async', async (req, res, next) => {
  try {
    const data = await fetchData()
    res.json(data)
  } catch (err) {
    next(err) // pasa al handler
  }
})
GET /error Lanza Error('esto falla') — activa el middleware 500 Probar →
GET /error500 Responde directamente con status 500 Probar →

Base de Datos – PostgreSQL

Pool de conexiones con pg · Consultas parametrizadas

Pool de conexiones

En lugar de abrir y cerrar una conexión por cada consulta, pg.Pool mantiene un grupo de conexiones reutilizables, mejorando el rendimiento bajo carga concurrente.

Configuración

Conectar a PostgreSQL

const { Pool } = require('pg')
const pool = new Pool({
  host:     process.env.DB_HOST,
  port:     process.env.DB_PORT,
  user:     process.env.DB_USER,
  password: process.env.DB_PASSWORD
})

Consulta parametrizada

SQL injection seguro

const result = await pool.query(
  'SELECT * FROM orders WHERE customer_id = $1',
  [req.params.clienteId]
)
res.json(result.rows)
$1, $2 → sin inyección SQL

GET /bdd/orders/:cliente Pedidos de un cliente (ej: /bdd/orders/1) Probar →
GET /bdd/orders/:cliente/:id Pedido concreto (ej: /bdd/orders/1/10248) Probar →

Blockchain – Web3.js

Ethereum · Infura · Balances · Bloques · BigInt

Web3.js v4 — cambio importante de importación

En Web3 v4 ya no se importa el módulo completo. Se usa desestructuración: const { Web3 } = require('web3'). Además, todos los valores numéricos de la blockchain son BigInt y deben convertirse antes de enviarse con res.json().

Configuración

Conectar a Infura

const { Web3 } = require('web3')
const web3 = new Web3(
  process.env.WEB3_PROVIDER_URL
)

BigInt Serialization

Helper para res.json()

const serializeBigInt = (obj) =>
  JSON.parse(JSON.stringify(obj,
    (_, v) => typeof v === 'bigint'
      ? v.toString() : v
  ))
BigInt → string

Object Storage – MinIO

Compatible con AWS S3 · Buckets · Objetos · URLs firmadas

Conceptos

Bucket vs Object

Bucket = contenedor raíz (como una carpeta). Object = cualquier fichero dentro de un bucket, identificado por su key (nombre/ruta).

Escala masivaURL firmadas

Iniciar MinIO

Comando PowerShell

$env:MINIO_ROOT_USER="minioadmin"
$env:MINIO_ROOT_PASSWORD="minioadmin"
.\minio.exe server C:\minio-data `
  --console-address :9001

vs fs local

¿Por qué MinIO?

A diferencia de fs (disco del servidor), MinIO ofrece URLs firmadas, políticas de acceso, replicación y es compatible con AWS S3, permitiendo migrar sin cambiar el código.

GET /minio/buckets Lista todos los buckets Probar →
POST /minio/buckets/:bucket Crear un nuevo bucket
GET /minio/buckets/:bucket/files Listar ficheros de un bucket
POST /minio/buckets/:bucket/upload Subir un fichero (multipart/form-data) Formulario →
GET /minio/buckets/:bucket/files/:filename/url URL de descarga firmada (presigned)
GET /minio/buckets/:bucket/files/:filename Descargar/recuperar un fichero
DELETE /minio/buckets/:bucket/files/:filename Eliminar un fichero (async/await)

CORS

Cross-Origin Resource Sharing · Same-Origin Policy · Preflight

¿Qué es CORS?

Los navegadores aplican la Same-Origin Policy: bloquean peticiones fetch/XHR a un dominio diferente al de la página. CORS permite al servidor relajar esta restricción añadiendo cabeceras HTTP que el navegador entiende.

Wildcard (todos)

CORS abierto

const cors = require('cors')
// Cualquier origen puede acceder
app.get('/datos', cors(), handler)
Access-Control-Allow-Origin: *

Origen específico

CORS restringido

const corsOpts = cors({
  origin: 'https://miapp.com'
})
app.get('/privado', corsOpts, handler)
Solo ese origen

Whitelist dinámica

Lista de orígenes

const lista = ['http://a.com', 'http://b.com']
cors({ origin: (orig, cb) => {
  cb(null, lista.includes(orig))
}})

Configuración completa

Opciones avanzadas

cors({
  origin:      'http://localhost:3333',
  methods:     ['GET','POST','DELETE'],
  credentials: true,
  maxAge:      86400
})
Preflight cacheCredentials
GET :3345/sin-cors Sin cabeceras CORS → el navegador bloquea
GET :3345/cors-abierto Access-Control-Allow-Origin: *
GET :3345/cors-restringido Solo acepta http://localhost:3333
POST :3345/cors-post Demuestra Preflight OPTIONS automático

Server Side Pages – Pug

Motor de plantillas · HTML dinámico generado en el servidor

✓ Diferencia clave

Archivos estáticos: el HTML está en disco y se sirve igual siempre. Plantillas Pug: el servidor genera el HTML en tiempo real combinando la plantilla con datos variables antes de enviárselo al cliente.

Sintaxis Pug

Sin corchetes ni etiquetas cierre

doctype html
html(lang='es')
  head
    title= titulo
  body
    h1= titulo
    p Bienvenido, #{nombre}
    ul
      each item in items
        li= item

Renderizado

Pasar variables al template

app.set('view engine', 'pug')

app.get('/t1', (req, res) => {
  res.render('t1', {
    titulo: 'Mi título',
    items:  ['a', 'b', 'c']
  })
})

Instalación

Paquete requerido

npm install pug

# Estructura de directorios:

  t1.pug
  usuario.pug
  layout.pug
npm i pug

Organización con Express Router

Dividir rutas en módulos · Escalabilidad · Trabajo en equipo

app.js ──► app.use('/api/clientes', ···) ──► routes/clientes.js
         ──► app.use('/api/productos', ···) ──► routes/productos.js
         ──► app.use('/api/facturas', ···) ──► routes/facturas.js

Estructura de un Router

routes/clientes.js

const router = express.Router()

router.get('/', handler)        // GET /api/clientes
router.get('/:id', handler)     // GET /api/clientes/:id
router.post('/', handler)       // POST /api/clientes
router.delete('/:id', handler)  // DELETE /api/clientes/:id

module.exports = { clientes: router }

Montar en app.js

app.use() con prefijo

const { clientes } = require('./routes/clientes')

// Todas las rutas del router quedan
// bajo el prefijo /api/clientes
app.use('/api/clientes', clientes)

Analizar la Performance

Benchmarking · Concurrencia · Tiempos de respuesta

¿Por qué medir el rendimiento?

Conocer cuántas peticiones por segundo puede manejar tu servidor, y cuáles rutas son más lentas, es esencial antes de poner una aplicación en producción. Node.js es no-bloqueante y puede manejar miles de conexiones concurrentes gracias al Event Loop.

SuperBenchmarker (Docker)

Benchmark con ab

# -v 2 → verbose, muestra detalles
docker run --rm jordi/ab -v 2 \
  http://192.168.1.40:3333/ping
Apache Benchmark

autocannon (Node.js)

Benchmark local

npx autocannon \
  -c 100 -d 10 \
  http://localhost:3333/ping

# -c 100 → 100 conexiones concurrentes
# -d 10  → 10 segundos de prueba
npm i -g autocannon

curl básico

Tiempo de respuesta

# Mide tiempo total de la petición
curl -o /dev/null -s \
  -w "Tiempo: %{time_total}s\n" \
  http://localhost:3333/ping

⚡ Tips de optimización

• Usa Promise.all() para llamadas asíncronas independientes en paralelo.
• Usa índices en PostgreSQL para consultas frecuentes.
• Cachea respuestas repetitivas (Redis, memoria) para rutas de solo lectura.
• Usa compression middleware para comprimir respuestas JSON grandes.
• En producción, usa un proxy inverso (Nginx) delante de Express.
