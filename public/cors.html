🌐 CORS – Cross-Origin Resource Sharing

Práctica completa: cómo el navegador controla las peticiones entre orígenes distintos

Esta página: http://localhost:3333 → fetch → Servidor CORS: http://localhost:3345
¿Qué es CORS? El navegador aplica la Same-Origin Policy: bloquea cualquier fetch o XHR que vaya a un origen diferente (protocolo + dominio + puerto distintos). El servidor puede autorizar orígenes específicos añadiendo la cabecera Access-Control-Allow-Origin.

Importante: CORS solo aplica en el navegador. curl, Postman y el REST Client no están sujetos a CORS.

Bloqueado por el navegador Servidor responde 200
El servidor responde correctamente, pero el navegador descarta la respuesta porque no hay Access-Control-Allow-Origin. Compruébalo en DevTools → Network.

Control de orígenes

Caso 3 — Solo un origen

GET http://localhost:3345/cors-restringido
Solo localhost:3333 Otros orígenes bloqueados
Access-Control-Allow-Origin: http://localhost:3333
El servidor comprueba el header Origin de la petición y solo responde con el permiso si coincide con el origen configurado. Patrón típico en producción para proteger tu API.

Caso 4 — Lista blanca de orígenes

GET http://localhost:3345/cors-whitelist
Lista de orígenes permitidos
Access-Control-Allow-Origin: <origen-validado>
Una función valida si el Origin está en la lista. Permite múltiples orígenes (frontend, móvil, staging…) sin usar *. Si el origen no está en la lista, el servidor devuelve un error.

Caso 6 — Preflight (petición OPTIONS)

POST http://localhost:3345/cors-post
Navegador envía OPTIONS primero Luego envía el POST real
Para peticiones "complejas" (POST con JSON, PUT, DELETE, cabeceras personalizadas) el navegador envía primero una petición OPTIONS automática para pedir permiso. Si el servidor responde correctamente, procede con la petición real. Puedes ver las dos peticiones en DevTools → Network.
